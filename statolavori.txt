1. OBIETTIVO DEL PROGETTO

---

Costruire una pipeline completamente offline e privacy first per trasformare fascicoli sanitari in dati strutturati utilizzabili per process mining e analisi cliniche.

Flusso logico finale desiderato:

PDF fascicolo sanitario
-> estrazione testo
-> estrazione strutturata tramite LLM locale o parser deterministico dedicato
-> validazione backend
-> salvataggio in strutture dati tracciabili
-> generazione event log
-> process mining
-> visualizzazione paziente singolo
-> visualizzazione per coorti o tipologie cliniche

Vincoli fondamentali:

* Nessuna chiamata esterna consentita
* LLM esclusivamente locale
* Separazione netta tra LLM e database
* Architettura scalabile a molti pazienti
* Privacy massima
* Ogni estrazione deve essere tracciabile
* Il testo e il layout originale devono poter essere auditati

Runtime LLM target attuale:

* Ollama locale
* Modello attualmente configurato: qwen3:30b-a3b

Importante sul significato di `127.0.0.1`

* Il modello non puo leggere da solo `reader_text.txt` o altri file del progetto solo perche gli passiamo il path
* E sempre il codice Python che deve leggere i file e passarne il contenuto al modello
* Questo resta vero sia con Ollama HTTP sia con un eventuale backend LLM in process

5. STATO SPERIMENTALE DEL CERTIFICATO VACCINALE

---

Problema reale osservato

* Il certificato vaccinale non e disposto come un elenco verticale pulito
* I vaccini sono stampati in righe con piu colonne orizzontali
* Le dosi hanno una coppia `numero dose - data`
* Sotto alcune colonne compaiono dettagli come nome prodotto, quantita e lotto
* Alcuni dettagli vanno a capo su piu righe
* Se si usa solo `extract_text()` il risultato perde la geometria del documento

Esempi concreti gia analizzati

* `MORBILLO`:
  * dose 2 -> `25/06/2007`
  * dose 1 -> `08/09/2003`
  * dettagli osservati sotto la colonna corretta:
    * `MMR II*SC 1FL + 1SIR.`
    * `0,5 ML`
    * `026265013`

* `POLIOMIELITE IPV`:
  * dose 3 -> `24/03/2003`
  * dose 5 -> `18/11/2024`
  * dose 4 -> `25/10/2004`
  * dose 2 -> `11/09/2002`
  * dose 1 -> `24/07/2002`
  * dettagli correttamente letti nello smoke test:
    * dose 5 -> `TRIAXIS`, `POLIO*SOSPENSIONE`, `INIETTABILE IN SIRINGA`
    * dose 4 -> `POLIOVAX-IN*IMSC 1 F`, `1 ML`, `034440014`

* `COVID 19`:
  * i dettagli possono ripetersi per piu dosi
  * questo e stato considerato accettabile dal punto di vista del dominio

Decisione pratica sul problema ambiguita

* I casi inizialmente marcati come ambigui sono stati rivalutati
* Se il blocco cade chiaramente sotto la riga sopra e una colonna domina nettamente, il parser puo trattarlo come corretto
* Non va introdotta ambiguita artificiale quando il layout reale suggerisce un associazione forte

6. COSA FA OGGI LO SMOKE TEST

---

File coinvolto

* `smoke_test_stages_1_to_5.py`

Comportamento attuale

* Legge un PDF reale dal dataset
* Classifica la famiglia documentale
* Estrae il testo lineare standard con `extract_text_from_pdf`
* Salva `extracted_text.txt`
* Se il documento e vaccinale, esegue un reader layout aware locale dentro lo smoke test
* Raggruppa parole in righe usando le coordinate verticali di `pdfplumber`
* Raggruppa le parole della riga in blocchi orizzontali
* Identifica le intestazioni vaccinali con pattern `dose - data`
* Costruisce zone orizzontali per ciascuna dose
* Assegna i blocchi di dettaglio alla dose piu plausibile usando le coordinate x
* Costruisce un `reader` intermedio che conserva i dettagli dose per dose
* Salva gli artefatti strutturati
* Costruisce un prompt dedicato ai vaccini
* Prova poi a chiamare Ollama in locale

Artefatti generati nello smoke test vaccinale

* `extracted_text.txt`
  testo piatto base, utile come controllo rapido
* `layout_text.txt`
  vista testuale del layout raggruppato per righe e blocchi
* `layout_words.json`
  artefatto tecnico con parole, coordinate, sezioni vaccinali e assegnazioni ai blocchi dose
* `reader_text.txt`
  rappresentazione leggibile del reader intermedio, pensata come input migliore per l LLM
* `reader.json`
  stessa informazione in forma JSON strutturata
* `llm_prompt.txt`
  prompt effettivamente passato o preparato per il modello
* `llm_response.json`
  risposta grezza e parsata del modello, quando disponibile

Percorso artefatti di test attuale

* `artifacts/pipeline_reale/CertificatoVaccinale_LMNLCU02E15D918M_20260303104746/`

Stato di esecuzione attuale

* Blocco 1 funziona e salva gli artefatti
* Il reader vaccinale sperimentale e attivo nello smoke test
* Il blocco LLM si ferma per timeout del modello locale

7. READER VACCINALE: SIGNIFICATO DEGLI ARTEFATTI

---

`extracted_text.txt`

* Serve solo per vedere il testo piatto estratto dal PDF
* Non e sufficiente per il parsing corretto dei vaccini

`layout_text.txt`

* Serve come vista umana piu leggibile del layout
* Non e la sorgente di verita per l estrazione clinica
* Da solo non basta per capire sempre a quale dose appartenga un dettaglio

`layout_words.json`

* E la sorgente tecnica piu importante del reader
* Conserva posizione e blocchi dei contenuti
* Permette di capire a quale colonna appartengono i dettagli
* E il punto giusto dove intervenire quando il layout viene caratterizzato male

`reader_text.txt`

* E un artefatto intermedio pensato per dare al modello un input piu pulito del testo piatto
* Non e il JSON clinico finale

`reader.json`

* E la versione strutturata del reader intermedio
* E utile sia per debug sia come possibile base di un builder deterministico del JSON clinico vaccinale

8. DECISIONE STRUTTURALE SUI VACCINI

---

Decisione emersa durante il lavoro

* Per i documenti vaccinali non e obbligatorio usare l LLM per costruire il JSON clinico finale
* Se il reader layout aware estrae gia bene dose, data e dettagli, ha senso costruire il JSON in modo deterministico
* Questa strada puo restare concettualmente nel blocco 2 della pipeline, ma come branch dedicato ai vaccini

Forma architetturale desiderata

* `stage1`:
  lettura PDF e reader layout aware vaccinale
* `stage2`:
  * per vaccini -> builder deterministico `reader -> clinical JSON`
  * per altri documenti -> branch LLM tradizionale
* `stage3`:
  salvataggio su database e artefatti
* `stage4`:
  derivazione eventi per process mining

Vantaggio

* Evitiamo di chiedere al modello di reinterpretare qualcosa che il parser ha gia capito bene
* Manteniamo comunque la stessa forma finale di output verso database e mining

9. PROPOSTA DI JSON FINALE PER IL DATABASE

---

Principio

* Il database deve ricevere fatti clinici normalizzati
* Il layout grezzo deve restare negli artefatti, non come dato clinico primario
* Ogni dose vaccinale deve poter diventare anche un evento clinico derivato

Shape proposto del JSON finale vaccinale

```json
{
  "document": {
    "source_document": "string",
    "document_family": "vaccination_certificate",
    "patient_id": "string or null",
    "parser": "vaccination_reader_v1",
    "parser_version": "string"
  },
  "vaccinations": [
    {
      "source_vaccine_label": "MORBILLO",
      "normalized_vaccine_key": "morbillo",
      "dose_number": 2,
      "administration_date": "2007-06-25",
      "product_name": "MMR II*SC 1FL + 1SIR.",
      "dose_amount_text": "0,5 ML",
      "lot_code": "026265013",
      "confidence": "high",
      "evidence_type": "exact",
      "ambiguity_notes": [],
      "source_detail_texts": [
        "MMR II*SC 1FL + 1SIR.",
        "0,5 ML",
        "026265013"
      ]
    }
  ],
  "clinical_events": [
    {
      "event_type": "vaccination",
      "activity_label": "Vaccination - MORBILLO",
      "occurred_at": "2007-06-25",
      "source_vaccine_label": "MORBILLO",
      "normalized_vaccine_key": "morbillo",
      "dose_number": 2,
      "confidence": "high",
      "source_document": "string"
    }
  ]
}
```

Campi ritenuti importanti

* `source_vaccine_label`
  nome come appare nel certificato
* `normalized_vaccine_key`
  chiave stabile per query e aggregazioni
* `dose_number`
  attributo esplicito, non nascosto nei tag
* `administration_date`
  data clinica della dose
* `product_name`, `dose_amount_text`, `lot_code`
  dettagli amministrativi o farmaceutici recuperati quando disponibili
* `confidence`
  qualita della lettura
* `evidence_type`
  `exact | ambiguous | missing`
* `source_detail_texts`
  audit rapido dei dettagli usati

13. TEMPLATE PROMPT LLM PER I VACCINI

---

Questo e il template concettuale da tenere per prove future sul modello. Va usato passando il contenuto del reader, non solo il path del file.

```text
You are extracting structured vaccination records from a layout-aware reader.
Return only valid JSON.
Never invent information.
If evidence is weak or ambiguous, keep uncertain fields null and explain ambiguity in notes.

Return one JSON object with this shape:
{
  "document_type": "vaccination_certificate",
  "source_document": "...",
  "vaccinations": [
    {
      "vaccine": "string",
      "dose_number": 1,
      "administration_date": "YYYY-MM-DD",
      "product_name": null,
      "dose_amount_text": null,
      "lot_code": null,
      "confidence": "high|medium|low",
      "evidence_type": "exact|ambiguous|missing",
      "ambiguity_notes": []
    }
  ]
}

Rules:
- Create one item for each dose that has a date.
- Use the vaccine name from the current VACCINE section.
- Convert dates from DD/MM/YYYY or DD-MM-YYYY to YYYY-MM-DD.
- Use exact details as the strongest evidence.
- If only ambiguous details exist, do not guess unsupported fields.
- If a dose exists but no reliable details are attached, still include it with null detail fields.
- Do not invent product names, lot codes, dose amounts, or routes.

Reader input:
[INCOLLARE QUI reader_text.txt]
```

Nota pratica

* Se si usa ancora l LLM, il prompt va costruito dal contenuto del reader
* Non basta dire al modello di leggere un file locale dal suo path
